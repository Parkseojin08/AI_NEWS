# AI News Hub 기획서

> 상세 구현 명세: [[PROJECT/AI_NEWS/기술설계 및 제작순서]]

## 1. 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트명 | AI News Hub (가칭) |
| 한 줄 정의 | Anthropic·OpenAI 공식 뉴스를 자동 수집하고 한국어 요약과 함께 한 화면에서 보는 개인용 웹 서비스 |
| 대상 사용자 | 본인 (개인용, 인증 없음 또는 단순 접근 제한) |
| 운영 비용 | 0원 (무료 티어만 사용) |

## 2. 배경 및 문제 정의

- Anthropic(anthropic.com/news)과 OpenAI(openai.com/news) 두 사이트를 매번 각각 방문해서 새 글을 확인하는 것이 번거로움
- 어디까지 읽었는지 추적이 안 되어 같은 목록을 반복해서 훑게 됨
- 영문 제목만으로는 클릭할 가치가 있는 글인지 판단이 어려움

## 3. 목표

1. 두 소스의 뉴스를 하나의 시간순 타임라인으로 통합
2. 새 글 등록 시 자동으로 한국어 3줄 요약 생성
3. 읽음/안읽음 상태 추적
4. 운영 비용 0원 유지

### 비목표 (Non-Goals)

- 댓글, 소셜, 다중 사용자 기능
- 기사 본문 전체 저장 (저작권 + 불필요. 원문 링크로 대체)
- 두 소스 외 언론사 뉴스 수집

## 4. 기능 정의

### 4.1 MVP (1차)

| 기능 | 설명 |
| --- | --- |
| 자동 수집 | RSS 피드를 주기적으로 파싱하여 신규 글만 DB 저장 |
| 통합 타임라인 | 두 소스 글을 발행일 내림차순으로 표시, 소스별 필터 토글 |
| 한국어 요약 | 신규 글의 제목+description을 LLM으로 3줄 요약 |
| 읽음 표시 | 카드 클릭(원문 이동) 시 읽음 처리, 안읽음 카운트 뱃지 |
| 수집 상태 표시 | 마지막 수집 시각을 UI에 노출 (수집기 침묵 사망 감지용) |

### 4.2 2차 (선택)

- 키워드 하이라이트 (예: "Claude Code", "SBOM" 포함 글 강조)
- Discord Webhook 알림
- 북마크 + Obsidian용 마크다운 클립 복사

## 5. 데이터 소스

| 소스 | 방식 | URL | 비고 |
| --- | --- | --- | --- |
| OpenAI | 공식 RSS | `https://openai.com/news/rss.xml` | 공식 제공, description 포함 |
| Anthropic | 커뮤니티 RSS | `https://raw.githubusercontent.com/Olshansk/rss-feeds/main/feeds/feed_anthropic_news.xml` | 매시간 스크래핑 생성. 외부 저장소 의존 |

- Anthropic은 공식 RSS 미제공. 커뮤니티 피드가 중단되면 anthropic.com/news 직접 스크래핑으로 교체 (Plan B)
- 한국어 지역화 페이지(openai.com/ko-KR/...)는 긁지 않음. 영문 RSS 수집 후 요약 단계에서 한국어로 출력

## 6. 기술 스택 및 아키텍처

### 6.1 스택

| 레이어 | 기술 | 비용 |
| --- | --- | --- |
| 프론트 | Vite + React (정적 빌드) on Vercel | 무료 |
| 백엔드 | Node.js + Express on Render Free Web Service | 무료 |
| 수집 트리거 | GitHub Actions schedule → Render 엔드포인트 호출 | 무료 |
| DB | Neon PostgreSQL Free | 무료 |
| 요약 LLM | Google Gemini API 무료 티어 (확정, 키 발급 완료) | 무료 |

**LLM 선정 근거**: Claude API는 무료 티어가 없어 "비용 0원" 제약과 충돌. Gemini API 무료 티어 사용으로 확정 — 구현 직전 현행 한도(분당/일일 요청 수)와 무료 모델명 확인. 무료 티어가 소멸했거나 제한이 과도하면 Fallback: RSS description 원문을 요약 없이 그대로 표시 (LLM 미사용, 완전 무료 보장).

### 6.2 아키텍처

```
GitHub Actions (schedule: 1시간 주기)
  → POST https://{app}.onrender.com/api/collect  (헤더: X-Cron-Secret)
    → OpenAI RSS + Anthropic 커뮤니티 RSS fetch/parse
    → url UNIQUE 기준 INSERT ... ON CONFLICT DO NOTHING
    → 신규 insert된 행만 Gemini 요약 → summary 컬럼 UPDATE
    → collect_log 기록

사용자 브라우저
  → Vercel (Vite 정적 빌드) → CORS → Render Express REST API
  → 카드 클릭 → PATCH /api/articles/:id/read → 원문 새 탭
```

- Actions 호출 자체가 잠든 Render를 깨우므로 스핀다운 상태에서도 수집이 동작함
- GitHub Actions schedule은 정시 실행 미보장 (수 분~수십 분 지연 가능). 뉴스 수집 용도로는 허용 범위

## 7. DB 설계 (Neon PostgreSQL)

```sql
CREATE TABLE articles (
  id            SERIAL PRIMARY KEY,
  source        VARCHAR(20) NOT NULL CHECK (source IN ('anthropic', 'openai')),
  title         TEXT NOT NULL,
  url           TEXT NOT NULL UNIQUE,
  description   TEXT,
  summary       TEXT,
  published_at  TIMESTAMPTZ NOT NULL,
  is_read       BOOLEAN NOT NULL DEFAULT FALSE,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_articles_published ON articles (published_at DESC);
CREATE INDEX idx_articles_source ON articles (source, published_at DESC);

CREATE TABLE collect_log (
  id            SERIAL PRIMARY KEY,
  source        VARCHAR(20) NOT NULL,
  fetched_count INT NOT NULL,
  new_count     INT NOT NULL,
  status        VARCHAR(10) NOT NULL CHECK (status IN ('ok', 'error')),
  error_message TEXT,
  executed_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

- 개인용 단일 사용자이므로 users 테이블 없이 `is_read`를 articles에 직접 보유
- `summary`가 NULL인 행 = 요약 실패/대기 상태. 다음 수집 사이클에서 재시도

## 8. API 설계

| 메서드 | 경로 | 설명 | 인증 |
| --- | --- | --- | --- |
| POST | `/api/collect` | RSS 수집 + 요약 실행 | X-Cron-Secret 헤더 |
| GET | `/api/articles` | 목록 조회 (`?source=&unread=&page=`) | 없음 |
| PATCH | `/api/articles/:id/read` | 읽음 처리 | 없음 |
| GET | `/api/status` | 마지막 수집 시각/결과 | 없음 |
| GET | `/health` | Render 헬스체크 + 프론트 웜업용 | 없음 |

## 9. 무료 티어 제약사항 및 대응

| 제약 | 내용 | 대응 |
| --- | --- | --- |
| Render Free 스핀다운 | 15분 유휴 시 스핀다운, 콜드 스타트 30~60초. 첫 API 응답 지연 | 프론트에 "서버 깨우는 중" 안내 + 페이지 로드 시 /health 선발사 웜업. 수집은 Actions 호출이 서버를 깨우므로 영향 없음 |
| Render Free 750시간/월 | 인스턴스 가동 시간 한도 | 단일 서비스 + 스핀다운 구조라 초과 불가능. 인지만 |
| GitHub Actions 60일 비활성화 | public 저장소에서 60일간 저장소 활동이 없으면 scheduled workflow 자동 비활성화. 완성 후 방치 시 확실히 걸림 | workflow에 keepalive 단계 추가 (주기적 더미 커밋 또는 keepalive 액션). 구현 필수 항목 |
| CRON_SECRET 이중 관리 | Render 환경 변수와 GitHub Actions Secret 두 곳에 동일 값 유지 필요. 한쪽만 변경 시 수집이 조용히 실패 | 변경 시 양쪽 동시 갱신 규칙화. 401 응답은 collect_log가 아닌 Actions 로그에서만 보이므로 Actions 실패 이메일 알림 켜두기 |
| Neon Free auto-suspend | 유휴 시 일시 정지, 첫 쿼리 지연 약 1초 내외 (추정) | 무시 가능. 수집 주기가 사실상 keep-alive 역할 |
| Gemini 무료 티어 rate limit | 분당 요청 수 제한 | 요약을 사이클당 최대 20건으로 제한. 429 시 다음 사이클 재시도 |
| Neon Free 저장 용량 | 수백 MB 수준 | 텍스트 메타데이터만 저장하므로 수년치도 문제없음 |
| 커뮤니티 RSS 중단 리스크 | Olshansk 저장소 유지보수 중단 가능 | collect_log에서 연속 실패 감지 → /api/status로 노출. Plan B: 직접 스크래핑 |

## 10. 환경 변수

| 변수 | 위치 | 용도 |
| --- | --- | --- |
| `DATABASE_URL` | Render | Neon 연결 문자열 |
| `CRON_SECRET` | Render + GitHub Secrets | /api/collect 인증 (양쪽 동일 값) |
| `GEMINI_API_KEY` | Render | 요약 LLM 키 (발급 완료) |
| `GEMINI_MODEL` | Render | 무료 티어 모델명 (구현 시 확정) |
| `CORS_ORIGIN` | Render | 허용 오리진 (Vercel 도메인) |
| `VITE_API_BASE_URL` | Vercel | Render API 주소 |
| `API_BASE_URL` | GitHub Secrets | Actions에서 호출할 Render 주소 |

모든 키는 Render/Vercel 환경 변수 + GitHub Actions Secrets로 관리. 코드 하드코딩 금지.
