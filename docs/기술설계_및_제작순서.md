# AI News Hub 기술설계 및 제작순서

> 이 문서 단독으로 구현 가능한 수준을 목표로 함. Claude Code sub agent에게 Phase 단위로 전달.
> 상위 문서: [[PROJECT/AI_NEWS/기획서]]

## 0. 기획서 대비 변경사항

| 항목 | 기획서 (§6) | 본 설계서 | 사유 |
| --- | --- | --- | --- |
| 프론트 | Next.js on Vercel | **Vite + React (정적 빌드) on Vercel** | 익숙한 스택 사용 |
| 백엔드 | Vercel Functions | **Node.js + Express on Render Free** | 익숙한 스택 사용 |
| 수집 트리거 | GitHub Actions → Vercel 함수 | GitHub Actions → Render 엔드포인트 | 동일 원리. Actions 호출이 잠든 Render를 깨우므로 스핀다운 문제 해소 |

주의: 이 변경으로 배포 구조가 기획서의 B안(Vercel 단일화)에서 A안(Render + Actions)으로 회귀함. 콜드 스타트(첫 API 응답 최대 1분)를 감수하는 선택. 기획서 §6은 추후 동기화 필요.

> **구현 편차 (2026-07-15, 메인 세션 승인)**: client 스캐폴드 언어를 JSX → **TypeScript**로 변경. §2 파일 트리의 확장자는 `.tsx`/`.ts` 기준이며, 컴포넌트/훅/api 래퍼 모두 TypeScript로 작성한다. build는 `tsc -b && vite build`. 사유: 실제 스캐폴드가 TS로 생성돼 있었고 타입 안정성 이점. Phase 5(frontend-agent)는 이 편차를 따른다.

> **구현 편차 (2026-07-16, 사용자 요청)**: server(백엔드)도 CommonJS JS → **TypeScript**로 전환. `server/src/*.ts`를 `tsc`로 `dist/`에 컴파일해 `node dist/index.js`로 실행. 응답 스키마·엔드포인트·로직은 불변(§6.3 계약 유지). `typescript`·`@types/*`는 dependencies에 배치(Render 빌드 대비). **배포 영향**: Render Build=`npm install && npm run build`, Start=`node dist/index.js`로 변경 필요(§9/DEPLOY.md 반영).

- 요약: Gemini API 무료 티어 (모델명은 구현 시점 최신 무료 모델 확인 후 `GEMINI_MODEL` 환경 변수로 주입)

## 5. DB 스키마 (migrations/001_init.sql)

```sql
CREATE TABLE IF NOT EXISTS articles (
  id            SERIAL PRIMARY KEY,
  source        VARCHAR(20) NOT NULL CHECK (source IN ('anthropic', 'openai')),
  title         TEXT NOT NULL,
  url           TEXT NOT NULL UNIQUE,
  description   TEXT,
  summary       TEXT,
  published_at  TIMESTAMPTZ NOT NULL,
  is_read       BOOLEAN NOT NULL DEFAULT FALSE,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_articles_published ON articles (published_at DESC);
CREATE INDEX IF NOT EXISTS idx_articles_source ON articles (source, published_at DESC);

CREATE TABLE IF NOT EXISTS collect_log (
  id            SERIAL PRIMARY KEY,
  source        VARCHAR(20) NOT NULL,
  fetched_count INT NOT NULL DEFAULT 0,
  new_count     INT NOT NULL DEFAULT 0,
  status        VARCHAR(10) NOT NULL CHECK (status IN ('ok', 'error')),
  error_message TEXT,
  executed_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

적용 방법: Neon 콘솔 SQL Editor에서 1회 실행 (마이그레이션 도구 불사용 — 테이블 2개 규모에 과잉).

## 6. 백엔드 상세 설계

### 6.1 데이터 소스

| 소스 | URL | 파싱 필드 |
| --- | --- | --- |
| openai | `https://openai.com/news/rss.xml` | title, link, description, pubDate |
| anthropic | `https://raw.githubusercontent.com/Olshansk/rss-feeds/main/feeds/feed_anthropic_news.xml` | title, link, description, pubDate |

정규화 규칙 (services/rss.js):
- `url`: link의 쿼리스트링(utm 등) 제거 후 저장 — 중복 방지 키이므로 필수
- `published_at`: pubDate 파싱 실패 시 수집 시각으로 대체하고 warn 로그
- `description`: HTML 태그 스트립, 2,000자 초과 시 절단
- title 또는 link 누락 항목은 skip

### 6.2 수집 플로우 (services/collector.js)

```
runCollect():
  for source in [openai, anthropic]:
    try:
      items = rss.fetchAndParse(source)        # 실패 시 이 소스만 error 로그, 다음 소스 계속
      newRows = INSERT ... ON CONFLICT (url) DO NOTHING RETURNING id, title, description
      collect_log에 (source, items.length, newRows.length, 'ok') 기록
    catch e:
      collect_log에 (source, 0, 0, 'error', e.message) 기록

  # 요약: summary IS NULL인 행 전부 대상 (이번 신규 + 과거 실패분 재시도)
  targets = SELECT id, title, description FROM articles
            WHERE summary IS NULL ORDER BY published_at DESC LIMIT 20
  for t in targets:
    try:
      s = summarizer.summarize(t.title, t.description)
      UPDATE articles SET summary = s WHERE id = t.id
    catch e:                                    # 429 포함 — 다음 사이클에 자동 재시도
      log.warn 후 계속 (트랜잭션 불필요, 행 단위 독립)
```

- LIMIT 20: Gemini 무료 티어 분당 한도 보호. 뉴스 발행량 기준 충분
- 요약 프롬프트: "다음 AI 기업 공식 뉴스의 제목과 설명을 보고 한국어 3문장으로 요약. 과장 금지, 사실만." + title + description
- 최초 수집 시 피드 과거분(수십 건)이 한 번에 들어옴 → 첫 2~3 사이클에 걸쳐 요약이 채워지는 것이 정상 동작

### 6.3 API 명세

#### POST /api/collect
- 헤더: `X-Cron-Secret: {CRON_SECRET}` — 불일치 시 401 (본문 없음, 로그만)
- 응답 200:
```json
{ "results": [
  { "source": "openai", "fetched": 25, "new": 2 },
  { "source": "anthropic", "fetched": 30, "new": 1 }
], "summarized": 3 }
```

#### GET /api/articles
- 쿼리: `source`(anthropic|openai, 생략 시 전체) / `unread`(true) / `page`(기본 1) / `limit`(기본 20, 최대 50)
- 정렬: published_at DESC
- 응답 200:
```json
{
  "items": [{
    "id": 1, "source": "anthropic", "title": "...",
    "url": "https://...", "summary": "...", "description": "...",
    "publishedAt": "2026-07-10T00:00:00Z", "isRead": false
  }],
  "page": 1, "totalPages": 5, "unreadCount": 12
}
```
- `summary`가 null이면 프론트는 description으로 대체 표시

#### PATCH /api/articles/:id/read
- 본문 없음. 응답 200 `{ "id": 1, "isRead": true }` / 존재하지 않는 id는 404

#### GET /api/status
- 응답 200:
```json
{ "lastCollect": [
  { "source": "openai", "executedAt": "...", "status": "ok", "newCount": 2 },
  { "source": "anthropic", "executedAt": "...", "status": "error", "errorMessage": "..." }
] }
```
- 소스별 최신 collect_log 1건씩

### 6.4 공통 규칙

- CORS: `CORS_ORIGIN` 목록만 허용 (콤마 구분 다중 지원)
- 에러 응답 포맷 통일: `{ "error": { "message": "..." } }` (스택트레이스 노출 금지)
- pg는 Pool 사용, 쿼리는 파라미터 바인딩($1) 필수 — SQL 인젝션 방지
- `GET /health` 추가: 200 반환만. Render 헬스체크 및 수동 웜업용

## 8. GitHub Actions (.github/workflows/collect.yml)

```yaml
name: collect-news
on:
  schedule:
    - cron: '0 * * * *'        # 매시 정각 (UTC, 지연 가능성 있음)
  workflow_dispatch:            # 수동 실행용

permissions:
  contents: write               # keepalive 커밋용

jobs:
  collect:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger collect endpoint
        run: |
          curl -sS -X POST "${{ secrets.API_BASE_URL }}/api/collect" \
            -H "X-Cron-Secret: ${{ secrets.CRON_SECRET }}" \
            --max-time 120 --retry 2 --retry-delay 30 --fail
      # Render 콜드 스타트 대비: --retry로 첫 요청 타임아웃 시 재시도

  keepalive:
    runs-on: ubuntu-latest
    if: github.event_name == 'schedule'
    steps:
      - uses: actions/checkout@v4
      - name: Keepalive commit (60일 비활성화 방지)
        run: |
          date > .github/keepalive
          git config user.name "keepalive-bot"
          git config user.email "actions@github.com"
          git add .github/keepalive
          git diff --cached --quiet || git commit -m "chore: keepalive [skip ci]"
          git push
```

- GitHub Secrets 등록: `API_BASE_URL`, `CRON_SECRET`
- keepalive는 매시 커밋이 과하면 별도 workflow로 분리해 월 1회 cron으로 조정 가능 (60일 조건만 회피하면 됨)

## 9. 배포 절차

1. **Neon**: 프로젝트 생성 → SQL Editor에서 001_init.sql 실행 → 연결 문자열 확보
2. **Render**: New Web Service → 레포 연결, Root Directory `server`, Build `npm install`, Start `node src/index.js` → 환경 변수 5종 입력 → 배포 후 `GET /health` 200 확인
3. **Vercel**: 프로젝트 import, Root Directory `client`, Framework Vite → `VITE_API_BASE_URL`에 Render URL 입력 → 배포
4. **Render CORS_ORIGIN**에 Vercel 도메인 반영 (재배포)
5. **GitHub Secrets** 등록 → Actions 탭에서 workflow_dispatch로 수동 1회 실행 → collect_log 확인

## 11. Sub Agent 구성 (.claude/agents/)

| Agent | 역할 | 전달할 문서 섹션 | 주요 규칙 |
| --- | --- | --- | --- |
| **backend-agent** | Express 서버, DB, 수집/요약 파이프라인, REST API | §2~6, §10 Phase 1~4 | 파라미터 바인딩 필수, env 하드코딩 금지, 소스 단위 에러 격리 |
| **frontend-agent** | Vite React UI 전체 | §2~4(client), §6.3(명세), §7, §10 Phase 5~6 | API 명세 임의 변경 금지(변경 필요 시 보고), 낙관적 업데이트 패턴 |
| **devops-agent** | GitHub Actions, Render/Vercel 설정, Secrets 문서화 | §1, §4, §8~9, §10 Phase 7 | 시크릿 값은 절대 파일에 기록하지 않고 등록 절차만 문서화 |
| **review-agent** | Phase 8 전체 리뷰 + 각 Phase 완료 기준 검수 | 문서 전체 + 완료 기준 표 | 완료 기준 미충족 시 해당 agent에 반려. 기존 code-review 스킬 재활용 |

- 메인 세션(orchestrator)이 Phase 순서 제어 및 완료 기준 판정, agent 간 산출물 전달
- CLAUDE.md에는 이 문서 경로와 "Phase 완료 기준을 통과하기 전 다음 Phase 착수 금지" 규칙 명시
